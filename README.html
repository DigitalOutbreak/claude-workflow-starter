Claude Workflow Starter

Drop-in setup for any new project. CLAUDE.md anchor, the five auto-imported context docs, the /feature lifecycle, the /cleanup housekeeping skill, and the code-scanner agent — wired up and ready to fill in.

Install git clone + bash setup.sh
Invoke /workflow-init
License MIT

What this is

A snapshot of the workflow that's running in digitaloutbreak/os, stripped of project-specific content and turned into a reusable starter. Run one script in a fresh project directory and you get the same context architecture: standing brief, layered context docs, feature lifecycle as slash commands, backlog discipline.

Three things to know up front:

  • Templates have placeholders. template files contain {{Project Name}} and bracketed prompts you fill in. The starter won't write your strategy memo for you — but it'll give you a structure that works.
  • Skills + agents are turnkey. copy as-is files work out of the box. /feature, /cleanup, and code-scanner need zero configuration.
  • Stack defaults can be edited. starter files (mainly coding-standards.md) assume Next 16 + TS + Tailwind v4. If your stack differs, swap the relevant sections.

Quick start

One command, from any new project directory: 

$ npx @digitaloutbreak/claude-workflow init

Or target an explicit path: 

$ npx @digitaloutbreak/claude-workflow init ./my-new-app

That's it. No clone, no folder structure assumptions, no PATH setup. npx caches the package; the installer refuses to overwrite if any target files exist.

Until the package is published to npm, run it straight from GitHub: 

$ npx github:DigitalOutbreak/claude-workflow-starter init

Optional: install the slash command

One-time per machine — adds /workflow-init to every Claude Code session: 

$ npx @digitaloutbreak/claude-workflow install

Then from any Claude Code session:

/workflow-init                  # install starter into current dir
/workflow-init ./my-new-app     # or specify a path

The slash command is a thin wrapper that calls the same npx … init. Skip the install if you'd rather always type the full npx command — the functionality is identical.

The script outputs each file it adds and ends with the four edits you should make next:

Conflict guard
If any target file already exists, the script lists every conflict and exits without writing anything. Move or delete the conflicts, then re-run. There's no --force flag on purpose — accidental overwrites of CLAUDE.md or current-feature.md would lose real work.

What to fill in

Three files determine whether the AI context is useful or generic. Spend an hour on these and the rest of the project will compound.

template CLAUDE.md
Replace {{Project Name}} and the project layout. Keep the @-imports and the commands list (edit the commands to match your scripts).
template thesis.md
The hardest one. Your founder memo — strategy, the bet, what you reject, the failure modes. Write it in your voice, not as a spec. 12 sections of scaffolding; delete the ones you don't need.
template project-overview.md
The polished summary that's auto-loaded every session. Scope, stack, surfaces, who uses it, decisions log. Diagram-heavy. This is the "if I had 10 minutes to onboard someone" doc.
template project-spec.md
Deeper authoritative detail — schema, RLS, env keys, webhook contracts. Read on demand only. Layered above overview, never auto-imported. Worth sketching upfront; rewrite as decisions get made.
starter coding-standards.md
Defaults assume Next 16 + TS + Tailwind v4. The opening note flags which sections to edit if your stack differs. Most of the file (naming, error handling, code quality) is stack-agnostic.
copy as-is ai-interaction.md
Workflow rules — branch naming, commit discipline, when to ask before changing direction, code review focus. Generic enough to leave alone.
copy as-is current-feature.md
Empty working file. Status / Goals / Notes / History. Fills up as you ship features. Don't pre-fill — let the first /feature load populate it.
copy as-is backlog.md
Empty index with category scaffolding (Performance, Mutations, Auth, etc.). Items get added by /feature complete from each shipped spec's "out of scope" list. Trust the workflow to populate it.

Your first feature

Once the strategy docs are filled in, the workflow takes over. Here's the first feature, end to end:

01
Open a Claude session in the project dir
CLAUDE.md auto-loads with the five @-imports. Tell Claude what you want to build.
02
/feature spec
Short interview → writes docs/context/features/<slug>-spec.md. The spec is the contract; everything downstream references it.
03
/feature load <slug>
Loads spec into current-feature.md — fills H1, Goals, Notes; sets Status = Not Started.
04
/feature start
Cuts feature/<slug> from main. Status = In Progress. Implementation begins.
05
Verify in browser + clean gates
Dev server, eyeball, run build/typecheck/lint. Iterate without committing.
06
/feature complete
Commit, merge to main with --no-ff, delete branch, reset current-feature.md, update backlog.md, push to origin. One command runs the entire close-out.
07
History grows; backlog tracks deferrals
A rich-prose entry lands at the bottom of current-feature.md § History. Anything explicitly deferred lands in backlog.md. The next session picks up with full context.

Feature lifecycle, reference

Command What it does Touches
/feature spec Short interview → writes a spec file docs/context/features/<slug>-spec.md
/feature load <slug> Loads spec into the working file docs/context/current-feature.md
/feature start Cuts branch, sets Status = In Progress git branch, current-feature.md
/feature review Checks goals met before completion reads only
/feature explain Documents what changed and why history prose
/feature complete Commit → merge → reset → backlog → push git, current-feature.md, backlog.md
/cleanup check Scans for console.logs, unused imports, TODOs, stale ignores reports only
/cleanup run Reports findings then asks which to fix code (with confirmation)
Invariant
Status always reflects reality: Not Started when loaded, In Progress when a branch exists, moved to History only when merged. Never commit without permission; never bypass build/typecheck/lint to ship.

Evolving the starter

The starter is a snapshot, not a framework. When you improve the workflow in a real project, copy the change back so the next project gets the upgrade. 

# Example: improved /feature complete in a real project
$ cp ~/projects/my-app/.claude/skills/feature/actions/complete.md \
     ~/Developer/_starters/claude-workflow/.claude/skills/feature/actions/complete.md

# Or take the entire feature skill
$ rsync -av --delete ~/projects/my-app/.claude/skills/feature/ \
     ~/Developer/_starters/claude-workflow/.claude/skills/feature/

# Version the starter properly
$ cd ~/Developer/_starters/claude-workflow
$ git init && git add . && git commit -m "snapshot"
$ gh repo create --private --source . --push    # optional: push to a private repo

The README and README.html are intentionally NOT copied by init.sh — they describe the starter, not whatever project you're starting.

What's NOT in here

Decided absences. Bring or build per project:

Missing Why not in the starter
Framework / package.json Bring your own stack. The starter is the workflow scaffold, not a Next.js template.
.env* files Per-project secrets. Use vercel env pull or equivalent.
Schema / migrations / seed scripts Domain-specific. The starter has no opinion about Postgres vs. SQLite vs. anything else.
UI components / shadcn primitives Install per project. The workflow doesn't require shadcn — that's a coding-standards default, not a hard dependency.
docs/handoffs/ Project-specific by definition. Create one when you need to brief another codebase.
The list-components skill Specific to the OS project's component layout. Not generic enough to ship in the starter.
